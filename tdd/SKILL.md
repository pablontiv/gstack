## AskUserQuestion Format

**ALWAYS follow this structure for every AskUserQuestion call:**
1. **Re-ground:** State the project, the current branch (use the `_BRANCH` value printed by the preamble — NOT any branch from conversation history or gitStatus), and the current plan/task. (1-2 sentences)
2. **Simplify:** Explain the problem in plain English a smart 16-year-old could follow. No raw function names, no internal jargon, no implementation details. Use concrete examples and analogies. Say what it DOES, not what it's called.
3. **Recommend:** `RECOMMENDATION: Choose [X] because [one-line reason]` — always prefer the complete option over shortcuts (see Completeness Principle). Include `Completeness: X/10` for each option. Calibration: 10 = complete implementation (all edge cases, full coverage), 7 = covers happy path but skips some edges, 3 = shortcut that defers significant work. If both options are 8+, pick the higher; if one is ≤5, flag it.
4. **Options:** Lettered options: `A) ... B) ... C) ...` — when an option involves effort, show both scales: `(human: ~X / CC: ~Y)`

Assume the user hasn't looked at this window in 20 minutes and doesn't have the code open. If you'd need to read the source to understand your own explanation, it's too complex.

Per-skill instructions may add additional formatting rules on top of this baseline.

## Completeness Principle — Boil the Lake

AI makes completeness near-free. Always recommend the complete option over shortcuts — the delta is minutes with CC+gstack. A "lake" (100% coverage, all edge cases) is boilable; an "ocean" (full rewrite, multi-quarter migration) is not. Boil lakes, flag oceans.

**Effort reference** — always show both scales:

| Task type | Human team | CC+gstack | Compression |
|-----------|-----------|-----------|-------------|
| Boilerplate | 2 days | 15 min | ~100x |
| Tests | 1 day | 15 min | ~50x |
| Feature | 1 week | 30 min | ~30x |
| Bug fix | 4 hours | 15 min | ~20x |

Include `Completeness: X/10` for each option (10=all edge cases, 7=happy path, 3=shortcut).

```bash
# Remote telemetry (opt-in, requires binary)
if [ "$_TEL" != "off" ] && [ -x ~/.claude/skills/gstack/bin/gstack-telemetry-log ]; then
  ~/.claude/skills/gstack/bin/gstack-telemetry-log \
    --skill "SKILL_NAME" --duration "$_TEL_DUR" --outcome "OUTCOME" \
    --used-browse "USED_BROWSE" --session-id "$_SESSION_ID" 2>/dev/null &
fi
```

Replace `SKILL_NAME` with the actual skill name from frontmatter, `OUTCOME` with
success/error/abort, and `USED_BROWSE` with true/false based on whether `$B` was used.
If you cannot determine the outcome, use "unknown". The local JSONL always logs. The
remote binary only runs if telemetry is not off and the binary exists.

## Plan Mode Safe Operations

When in plan mode, these operations are always allowed because they produce
artifacts that inform the plan, not code changes:

- `$B` commands (browse: screenshots, page inspection, navigation, snapshots)
- `$D` commands (design: generate mockups, variants, comparison boards, iterate)
- `codex exec` / `codex review` (outside voice, plan review, adversarial challenge)
- Writing to `~/.gstack/` (config, analytics, review logs, design artifacts, learnings)
- Writing to the plan file (already allowed by plan mode)
- `open` commands for viewing generated artifacts (comparison boards, HTML previews)

These are read-only in spirit — they inspect the live site, generate visual artifacts,
or get independent opinions. They do NOT modify project source files.

## Skill Invocation During Plan Mode

If a user invokes a skill during plan mode, that invoked skill workflow takes
precedence over generic plan mode behavior until it finishes or the user explicitly
cancels that skill.

Treat the loaded skill as executable instructions, not reference material. Follow
it step by step. Do not summarize, skip, reorder, or shortcut its steps.

If the skill says to use AskUserQuestion, do that. Those AskUserQuestion calls
satisfy plan mode's requirement to end turns with AskUserQuestion.

If the skill reaches a STOP point, stop immediately at that point, ask the required
question if any, and wait for the user's response. Do not continue the workflow
past a STOP point, and do not call ExitPlanMode at that point.

If the skill includes commands marked "PLAN MODE EXCEPTION — ALWAYS RUN," execute
them. The skill may edit the plan file, and other writes are allowed only if they
are already permitted by Plan Mode Safe Operations or explicitly marked as a plan
mode exception.

Only call ExitPlanMode after the active skill workflow is complete and there are no
other invoked skill workflows left to run, or if the user explicitly tells you to
cancel the skill or leave plan mode.

# Test-Driven Development

## Iron Law

**NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST.**

This is not a guideline. This is not a suggestion. This is an invariant. Every line
of production code exists because a test demanded it. If you catch yourself writing
production code before a failing test, STOP, delete the production code, and start
over with the test.

Why: code without a failing test is untested by definition. "I'll add tests after"
means "I'll verify my assumptions using the same brain that made those assumptions."
Tests written after the fact test what the code does, not what it should do. TDD
forces you to specify the behavior before implementing it.

---

## Step 0: Setup

Parse `$ARGUMENTS` for the feature or behavior to implement.

**Read project conventions:**

```bash
[ -f "CLAUDE.md" ] && head -100 CLAUDE.md
```

Look for: test commands, test file naming conventions, test directory structure,
preferred assertion libraries. If CLAUDE.md specifies a test command, use that
for ALL test runs throughout this session.

**Auto-detect test framework (only if CLAUDE.md does not specify):**

```bash
[ -f "go.mod" ] && echo "FRAMEWORK: go test ./..."
[ -f "Cargo.toml" ] && echo "FRAMEWORK: cargo test"
[ -f "pyproject.toml" ] && echo "FRAMEWORK: pytest"
[ -f "mix.exs" ] && echo "FRAMEWORK: mix test"
[ -f "build.gradle" ] || [ -f "build.gradle.kts" ] && echo "FRAMEWORK: gradle test"
[ -f "pom.xml" ] && echo "FRAMEWORK: mvn test"
[ -f "Gemfile" ] && echo "FRAMEWORK: bundle exec rspec"
[ -f "deno.json" ] || [ -f "deno.jsonc" ] && echo "FRAMEWORK: deno test"
```

```bash
[ -f "package.json" ] && cat package.json | grep -E '"(jest|vitest|mocha|ava|tap|bun)"|"test"' || true
```

If no framework is detected and no test command exists in CLAUDE.md, use
AskUserQuestion: "No test framework detected. What command runs your tests?
(e.g., `go test ./...`, `bun test`, `pytest`)"

Remember the detected test command as TEST_CMD for the rest of this session.
Reference it in prose as "the test command detected in Step 0."

**Check for existing test patterns:**

```bash
find . -name "*test*" -o -name "*spec*" | head -20
```

Read 1-2 existing test files to match the project's style: file naming, imports,
assertion style, describe/it nesting, setup/teardown patterns. The tests you write
must look like they belong in this codebase.

## Step 1: Identify Test Cases

**Check for Gherkin feature files first:**

```bash
ls specs/features/*.feature 2>/dev/null || ls features/*.feature 2>/dev/null || ls spec/features/*.feature 2>/dev/null || echo "NO_FEATURE_FILES"
```

1. If `.feature` files exist: read the Gherkin scenarios. Each `Scenario:` or
   `Scenario Outline:` becomes one test case. Extract the behavior described in
   the Given/When/Then steps. Preserve the scenario names for test descriptions.

2. If no `.feature` file exists: decompose the user's requirement into individual
   behaviors. Each behavior is one testable unit. Think about:
   - Happy path (the main success scenario)
   - Edge cases (empty input, boundary values, null/nil)
   - Error cases (invalid input, missing dependencies, network failures)
   - State transitions (before/after, side effects)

3. Present the test plan to the user via AskUserQuestion:

   ```
   TEST PLAN
   ════════════════════════════════════════
   Source: [.feature file path | user description]

   Test cases (in implementation order):
   1. [behavior description] — [why this matters]
   2. [behavior description] — [why this matters]
   3. ...

   Total: N test cases
   ════════════════════════════════════════

   Proceed with this plan?
   A) Yes, start the RED-GREEN-REFACTOR cycles
   B) Modify — I want to add/remove/reorder test cases
   C) Abort
   ```

---

## Step 2: RED -- Write a Failing Test

For the current test case:

1. Write ONE test that describes the expected behavior. The test name should read
   like a specification: "returns empty list when no items match filter", not
   "test filter" or "test1".

2. Run it with the test command detected in Step 0:

   ```bash
   # Run just the new test file (faster feedback)
   <test-command> <test-file>
   ```

3. **Verify the test FAILS.** This is the RED in RED-GREEN-REFACTOR.

4. **Verify it fails for the RIGHT REASON.** The failure must be about the missing
   behavior, not about:
   - Syntax errors in the test itself
   - Missing imports or unresolved modules
   - Wrong file path or test configuration
   - Compilation errors in the test

   If the failure is a test infrastructure issue, fix the test setup first
   until the test fails because the production behavior does not exist yet.

5. If the test PASSES without writing any production code: the behavior already
   exists. Log this: "SKIP: [test case] — behavior already implemented." Move to
   the next test case in Step 5.

---

## Step 3: GREEN -- Minimal Implementation

1. Write the MINIMUM code to make the failing test pass. This means:
   - No extra methods "we'll need later"
   - No abstractions "for future flexibility"
   - No handling of cases not yet tested
   - Hardcoding a return value is acceptable if only one test exists for that path

2. Run the failing test again:

   ```bash
   <test-command> <test-file>
   ```

   Verify it passes. If it still fails, read the error carefully and fix the
   production code. Do NOT modify the test to make it pass — the test defines the
   correct behavior.

3. Run ALL tests to catch regressions:

   ```bash
   <test-command>
   ```

   If any other test breaks, the new code introduced a regression. Fix it before
   proceeding. The fix must not break the new test.

---

## Step 4: REFACTOR

Now that all tests pass, clean up both test and production code:

1. Look for:
   - Duplicated code that can be extracted
   - Names that don't communicate intent
   - Functions that do too many things
   - Test setup that can be shared via helpers or fixtures

2. Make ONE refactoring change at a time. After each change, run ALL tests:

   ```bash
   <test-command>
   ```

3. If tests fail after a refactoring step: undo that specific change immediately.
   The refactoring was not behavior-preserving. Try a smaller, safer refactoring
   instead.

4. Refactoring is optional per cycle. If the code is already clean after GREEN,
   move on. Do not refactor for the sake of refactoring.

---

## Step 5: Next Cycle

Move to the next test case from the plan in Step 1. Return to Step 2.

Repeat Steps 2-4 for each test case. After each cycle, briefly note the status:

```
CYCLE N: [test case description]
  RED:      test fails (correct reason: [reason])
  GREEN:    test passes, all tests pass
  REFACTOR: [what was cleaned up, or "no refactoring needed"]
```

After ALL test cases are complete, run the full test suite one final time:

```bash
<test-command>
```

All tests must pass. If any fail, debug and fix before proceeding to completion.

---

## Rationalization Detection

STOP IMMEDIATELY if you catch yourself thinking any of these:

| Thought | Reality |
|---------|---------|
| "Let me just write the code first, I'll add tests after" | That is not TDD. Write the test first. |
| "This is too simple to need a test" | Simple code still breaks. Write the test. |
| "I know this works, the test is just a formality" | Then the test will pass quickly. Write it first. |
| "Let me scaffold the structure first" | Structure emerges from tests. Start with the test. |
| "I'll write multiple tests at once to save time" | One test at a time. Focus. |
| "The test framework isn't set up" | Setting it up IS your first task. Do it. |
| "Let me refactor before writing the next test" | Refactoring happens in Step 4, after GREEN. |
| "This edge case is obvious, I'll just handle it" | If it is obvious, the test will be trivial to write. Write it. |

**If you wrote production code before a failing test:** DELETE the production code.
Return to Step 2 and write the test first. No exceptions.

## Completion

After all cycles are complete and the full test suite passes, output a structured report:

```
TDD REPORT
════════════════════════════════════════
Feature:     [what was implemented]
Source:      [.feature file path, or "user description"]
Tests:       N written, N passing, N skipped (already existed)
Cycles:      N RED-GREEN-REFACTOR cycles completed
Test files:  [list of test files created/modified]
Source files:[list of production files created/modified]
Status:      DONE | DONE_WITH_CONCERNS | BLOCKED
════════════════════════════════════════
```

**Status definitions:**
- **DONE:** All test cases implemented, all tests pass, code is clean.
- **DONE_WITH_CONCERNS:** All tests pass but something warrants attention (e.g., a test
  was hard to write suggesting a design issue, or an edge case was deferred).
- **BLOCKED:** Cannot proceed. Test framework issue, unclear requirements, or a
  dependency is missing. Explain what is blocking and what the user needs to do.

## Capture Learnings

If you discovered a non-obvious pattern, pitfall, or architectural insight during
this session, log it for future sessions:

```bash
~/.claude/skills/gstack/bin/gstack-learnings-log '{"skill":"tdd","type":"TYPE","key":"SHORT_KEY","insight":"DESCRIPTION","confidence":N,"source":"SOURCE","files":["path/to/relevant/file"]}'
```

**Types:** `pattern` (reusable approach), `pitfall` (what NOT to do), `preference`
(user stated), `architecture` (structural decision), `tool` (library/framework insight),
`operational` (project environment/CLI/workflow knowledge).

**Sources:** `observed` (you found this in the code), `user-stated` (user told you),
`inferred` (AI deduction), `cross-model` (both Claude and Codex agree).

**Confidence:** 1-10. Be honest. An observed pattern you verified in the code is 8-9.
An inference you're not sure about is 4-5. A user preference they explicitly stated is 10.

**files:** Include the specific file paths this learning references. This enables
staleness detection: if those files are later deleted, the learning can be flagged.

**Only log genuine discoveries.** Don't log obvious things. Don't log things the user
already knows. A good test: would this insight save time in a future session? If yes, log it.

---

## Important Rules

1. **ONE test at a time.** Never write multiple tests before making them pass.
2. **The test MUST fail before you write production code.** No exceptions.
3. **Run ALL tests after every GREEN and every REFACTOR step.** Regressions hide.
4. **If a test is hard to write, the design needs to change.** That is valuable
   feedback, not an obstacle. Difficulty writing a test means the code is too
   coupled, has hidden dependencies, or the interface is wrong.
5. **Minimal GREEN.** Write the least code to pass. Resist the urge to "finish"
   the implementation. The next test will drive the next increment.
6. **benefits-from: investigate.** If a test failure is mysterious or the root cause
   is unclear, invoke `/investigate` to systematically debug it before guessing.
7. **Match project style.** Test file names, assertion patterns, describe/it nesting,
   and setup/teardown must match what already exists in the codebase.
8. **Do not modify existing tests** unless the requirement explicitly changes their
   expected behavior. Existing tests are a safety net, not an obstacle.
